NetCDF Dataset

John Caron

last changed: Aug 22, 2003

Introduction

A netCDF dataset is a generalization of a NetCDF file. Its purpose is to allow:

  1. Metadata to be added, deleted, changed and in general redefined.
  2. Variables to be renamed, added, deleted and restructured.
  3. Information from multiple NetCDF files to be combined.
  4. Transparent access from clients, ie it provides the usual NetCDF API.

A netCDF dataset is defined by an XML document called the netCDF Dataset Definition. This uses the netcdfDataset.xsd schema which is an extension of the base netcdf.xsd schema.

Currently, a netCDF dataset is accessed through the ucar.nc2.dataset package of the Java netCDF library (version 2.1). The abstractions and XML syntax defined here are language independent, and in the future a C library implementation could be created to provide equivalent functionality.

Also see:

  • NetCDF Markup Language (NcML)
  • NetCDF Coordinate Systems
  • Aggregating netCDF files with netCDF Datasets.

Definitions

NetCDF File: A physical netCDF file. This can be accessed as a file, through HTTP, or through the OpenDAP (aka DODS) protocol

NetCDF Dataset: A generalization of a NetCDF file, defined by an XML document called the NetCDF Dataset Definition.

NetCDF Dataset Definition. The Dataset Definition completely specifies the NetCDF Dataset, overiding all metadata in the NetCDF file itself. The only reason to access the file is if a variable's values are not declared. The URL of the NetCDF Dataset is the NetCDF Dataset Definition XML document.

NetCDF Dataset Metadata: All the attributes, dimensions, and variables, and optionally some variable data, of a NetCDF Dataset. Equivailent to the netCDF header, which is the output of the "ncdump -h" command, except that it may also include variable data (like what is often in a CDL file). It is all of the information specified in the NetCDF Dataset Definition.

NetCDF Dataset Data: The NetCDF dataset variable data.

NetCDF Schema Specification: An XML language for specifying the metadata in a NetCDF file. See http://www.unidata.ucar.edu/schemas/netcdfDataset.xsd

Examples

1. Completely specify the NetCDF Dataset Metadata:

In this example, all of the metadata is specified in the XML, and the underlying file is used only for data access. If there are any other attributes, variables or dimensions in the file, they are ignored.

<?xml version="1.0" encoding="UTF-8"?>
<nc:netcdf xmlns:nc="http://www.ucar.edu/schemas/netcdf"
  xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
  xsi:schemaLocation="http://www.ucar.edu/schemas/netcdf http://www.unidata.ucar.edu/schemas/netcdfDataset2.xsd" 	
  uri="file:///E:/dev/NCdataset/example.nc">
  
  <nc:dimension name="time" length="2" isUnlimited="true"/>
  <nc:dimension name="lat" length="3"/>
  <nc:dimension name="lon" length="4"/>

  <nc:variable name="rh" type="int" shape="time lat lon">
    <nc:attribute name="long_name" type="string" value="relative humidity"/>
    <nc:attribute name="units" type="string" value="percent"/>
  </nc:variable>
	
  <nc:variable name="T" type="double" shape="time lat lon">
    <nc:attribute name="long_name" type="string" value="surface temperature"/>
    <nc:attribute name="units" type="string" value="degC"/>
  </nc:variable>

  <nc:variable name="lat" type="float" shape="lat">
    <nc:values separator="">41.0 40.0 39.0</nc:values>
    <nc:attribute name="units" type="string" value="degrees_north"/>
  </nc:variable>
  
  <nc:variable name="lon" type="float" shape="lon">
    <nc:values separator="">-109.0 -107.0 -105.0 -103.0</nc:values>
    <nc:attribute name="units" type="string" value="degrees_east"/>
  </nc:variable>
  
  <nc:variable name="time" type="int" shape="time">
    <nc:values separator="">6 18</nc:values>
    <nc:attribute name="units" type="string" value="hours"/>
  </nc:variable>
  
  <nc:attribute name="title" type="string" value="Example Data"/>
  <nc:attribute name="testByte" type="byte" value="1 2 3 4"/>
  <nc:attribute name="testShort" type="short" value="1 2 3 4"/>
  <nc:attribute name="testInt" type="int" value="1 2 3 4"/>
  <nc:attribute name="testFloat" type="float" value="1 2 3 4"/>
  <nc:attribute name="testDouble" type="double" value="1 2 3 4"/>
</nc:netcdf>

Data for the lat, lon and time coordinate variables comes from the XML document, while the data for the rh and T variables will be obtained from the netCDF file at URL file:///E:/dev/NCdataset/example.nc. Note that in the CDL that follows, the netCDF name is the name of the XML dataset definition, not the underlying netCDF file specified in the uri attribute. This creates the following NetCDF Dataset, in CDL notation:

netcdf file:///E:/dev/NCdataset/xml/testRead.xml {

 dimensions:
  time = UNLIMITED;   // (2 currently)   // (has coord.var)
  lat = 3;   // (has coord.var)
  lon = 4;   // (has coord.var)

 variables:
  int rh(time, lat, lon);
   :long_name = "relative humidity";
   :units = "percent";
  double T(time, lat, lon);
   :long_name = "surface temperature";
   :units = "degC";
  float lat(lat);
   :units = "degrees_north";
  float lon(lon);
   :units = "degrees_east";
  int time(time);
   :units = "hours";
 // Global Attributes:
  :title = "Example Data";
  :testByte = 1, 2, 3, 4; // byte
  :testShort = 1, 2, 3, 4; // short
  :testInt = 1, 2, 3, 4; // int
  :testFloat = 1.0, 2.0, 3.0, 4.0; // float
  :testDouble = 1.0, 2.0, 3.0, 4.0; // double
 data:
  lat = 41.0, 40.0, 39.0;
  lon = -109.0, -107.0, -105.0, -103.0;
  time = 6, 18;
}

2. Use the metadata in the existing file:

In this example, all of the metadata is read in from the underlying file. 

<netcdf xmlns="http://www.ucar.edu/schemas/netcdf" 
    xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
	xsi:schemaLocation="http://www.ucar.edu/schemas/netcdf http://www.unidata.ucar.edu/schemas/netcdfDataset2.xsd"
	uri="file:///E:/dev/NCdataset/example.nc">

  <readMetadata/>
</netcdf>

Here, the <readMetadata/> element causes all of the metadata from the netCDF file to be read in. This creates the following NetCDF Dataset: 

netcdf file:///E:/dev/NCdataset/xml/useExisting.xml {
  dimensions:
    time = UNLIMITED;   // (2 currently)   // (has coord.var)
	lat = 3;   // (has coord.var)
	lon = 4;   // (has coord.var)
  variables:
    int rh(time, lat, lon);
	  :long_name = "relative humidity";
	  :units = "percent";
	double T(time, lat, lon);
	  :long_name = "surface temperature";
	  :units = "degC";
	float lat(lat);
	  :units = "degrees_north";
	float lon(lon);
	  :units = "degrees_east";
	int time(time);
	  :units = "hours";
	
  // Global Attributes:
    :title = "Example Data";
}
(Note that the namespace declaration in the above netcdf element xmlns="http://www.ucar.edu/schemas/netcdf" makes the netcdf namespace the default namespace, so we do not need to qualify the element names with the nc: prefix.)

3. Read in metadata from existing NetCDF file and override:

Here we read in all of the metadata from the underlying file, and modify it through elements in the XML:

<?xml version="1.0" encoding="UTF-8"?>
<nc:netcdf xmlns:nc="http://www.ucar.edu/schemas/netcdf"
    xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
    xsi:schemaLocation="http://www.ucar.edu/schemas/netcdf http://www.unidata.ucar.edu/schemas/netcdfDataset2.xsd" 
    uri="file:///E:/dev/NCdataset/example.nc">

  <nc:attribute name="Conventions" type="string" value="added"/>
  <nc:attribute name="title" type="string" value="replaced"/>
  
  <nc:dimension name="lat" length="3"/>
  <nc:variable name="deltaLat" type="double" shape="lat">
    <nc:values>.1 .1 .01</nc:values>
  </nc:variable>

  <nc:readMetadata>
    <nc:rename type="variable" rename="Temperature" nameInFile="T"/>
    <nc:readVariable name="rh" rename="ReletiveHumidity">
      <nc:attribute name="long_name2" type="string" value="relatively humid"/>
      <nc:attribute name="units" type="string" value="percent (%)"/>
      <nc:removeAttribute name="long_name"/>
      <nc:renameAttribute rename="checkedBy" nameInFile="QA" />
    </nc:readVariable >
  </nc:readMetadata>
</nc:netcdf>

The <nc:readMetadata> element again reads all of the metadata from the named file, but any elements that are already decalred in the XML override it. The global attribute "Conventions" is added, and the global attribute "title" replaces the attribute from the existing file. (Its not obvious what is added and what is modified unless you know what's in the underlying file.)

A new variable "deltaLat" is added, and since it refers to the lat dimension in its shape attribute, that dimension must be declared here. Since there is no corresponding variable in the netCDF file, its values must be specified here also.

Inside the <nc:readMetadata> element are elements which modify or refer to metadata read in by the<nc:readMetadata>. The variable "T" is renamed to "Temperature". The nc:readVariable element declares that the variable "rh" will be read in, and modifies its attributes. It also renames the variable to "ReletiveHumidity". The variable attribute "long_name2" is added, "units" is replaced, "long_name" is removed, and "QA" is renamed to "checkedBy".

This creates the following NetCDF Dataset. 

netcdf file:///E:/dev/NCdataset/xml/replace.xml {
  dimensions:
    time = UNLIMITED;   // (2 currently)   // (has coord.var)
    lat = 3;   // (has coord.var)
    lon = 4;   // (has coord.var)

  variables:
    double deltaLat(lat);
    int ReletiveHumidity(time, lat, lon);
      :checkedBy = "number 92";
      :long_name2 = "relatively humid";
      :units = "percent (%)";
    double Temperature(time, lat, lon);
      :long_name = "surface temperature";
      :units = "degC";
    float lat(lat);
      :units = "degrees_north";
    float lon(lon);
      :units = "degrees_east";
    int time(time);
      :units = "hours";
	  
  // Global Attributes:
  :Conventions = "added";
  :title = "replaced";

  data:
    deltaLat = 0.1, 0.1, 0.01;
}

4. Completely specify the NetCDF Dataset Metadata, rename variables.

Here, all of the metadata is specified in the XML, and the data for "ReletiveHumidity" variable will be taken from the variable named "rh" in the file. This is done by adding the "nameInFile" attribute to the variable element.

<?xml version="1.0" encoding="UTF-8"?>
<netcdf xmlns="http://www.ucar.edu/schemas/netcdf"
    xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
    xsi:schemaLocation="http://www.ucar.edu/schemas/netcdf http://www.unidata.ucar.edu/schemas/netcdfDataset2.xsd" 	
    uri="file:///E:/dev/NCdataset/example.nc">

  <dimension name="time" length="2" isUnlimited="true"/>
  <dimension name="lat" length="3"/>
  <dimension name="lon" length="4"/>
  
  <variable name="ReletiveHumidity" type="int" shape="time lat lon" nameInFile="rh">
    <attribute name="long_name" type="string" value="relative humidity"/>
    <attribute name="units" type="string" value="percent"/>
  </variable>
<netcdf>


5. Subset a variable's data

In this example xml, we use logicalView elements to define new variables (T-FHAG, T-MandatoryLevels, and T-BoundaryLayers) whose data are subsets of an existing variable (named t in the file). We do this because the original file has 3 vertical coordinates munged together in the levels_25 dimension, and we want to break that into 3 seperate variables.

The locicalView element currrently has only one attribute section whose value defines the data section, according to these rules:

  • to select all of a dimension, use as asterisk (*)
  • to select one slice, just use the value of the index
  • to select a range, use start:end, eg 1:19
<?xml version="1.0" encoding="UTF-8"?>
<netcdf xmlns="http://www.ucar.edu/schemas/netcdf" 
	xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
	xsi:schemaLocation="http://www.ucar.edu/schemas/netcdf http://www.unidata.ucar.edu/schemas/netcdfDataset2.xsd"
	uri="file:///C:/data/conventions/awips/awipsTestFilename.nc">
 <dimension name="record" length="11" isUnlimited="true"/>
 <dimension name="x" length="93"/>
 <dimension name="y" length="65"/>
 <dimension name="levels_25" length="25"/>
 <dimension name="FHAG" length="1"/>
 <dimension name="MandatoryLevels" length="19"/>
 <dimension name="BoundaryLayers" length="5"/>
 
 <variable name="T-FHAG" shape="record FHAG y x" type="float"    nameInFile="t">
   <attribute name="long_name" type="string" value="Temperature    at fixed height above ground"/>
   <attribute name="units" type="string" value="K"/>
   <attribute name="valid_range" type="float" value="180.0    330.0"/>
   <attribute name="_FillValue" type="float" value="-99999.0"/>
   <logicalView section="*,0,*,*" />
 </variable>
   
 <variable name="T-MandatoryLevels" shape="record MandatoryLevels    y x" type="float" nameInFile="t">
   <attribute name="long_name" type="string" value="Temperature    on Mandatory Levels"/>
   <attribute name="units" type="string" value="K"/>
   <attribute name="valid_range" type="float" value="180.0    330.0"/>
   <attribute name="_FillValue" type="float" value="-99999.0"/>
   <logicalView section="*,1:19,*,*" />
 </variable>
   
 <variable name="T-BoundaryLayers" shape="record BoundaryLayers    y x" type="float" nameInFile="t">
   <attribute name="long_name" type="string" value="Temperature    on BoundaryLayers"/>
   <attribute name="units" type="string" value="K"/>
   <attribute name="valid_range" type="float" value="180.0    330.0"/>
   <attribute name="_FillValue" type="float" value="-99999.0"/>
   <logicalView section="*,20:24,*,*" />
 </variable>
   
 <variable name="t" shape="record levels_25 y x" type="float">
   <attribute name="long_name" type="string" value="Temperature"/>
   <attribute name="units" type="string" value="K"/>
   <attribute name="valid_range" type="float" value="180.0    330.0"/>
   <attribute name="_FillValue" type="float" value="-99999.0"/>
 </variable>
   
 <variable name="FHAG" type="float" shape="FHAG">
   <attribute name="long_name" type="string" value="fixed height above ground"/>
   <attribute name="units" type="string" value="m"/>
   <attribute name="positive" type="string" value="up"/>
   <values>2.0</values> 
 </variable>
   
 <variable name='MandatoryLevels' type="int" shape="MandatoryLevels">
   <attribute name="units" type="string" value="hPa"    />
   <attribute name="long_name" type="string" value="Pressure    Levels" />
   <values>1000 950 900 850 800 750 700 650 600 550 500 450 400 350 300 250    200 150 100</values> 
 </variable>
   
 <variable name="BoundaryLayers" shape="BoundaryLayers"    type="double">
   <attribute name="units" type="string" value="hPa"    />
   <attribute name="long_name" type="string" value="BoundaryLayer    hectoPascals above ground" />
   <values start="0.0" increment="30.0" npts="5"    />
  </variable >
   
</netcdf>

NB: This example is taken from the infamous AWIPS formatted netCDF files, and is a good example of using the logicalView element. If you have to work with AWIPS files, the dataset convention parsing code will do all the hard work, so you do not need to use NcML to break up the variables yourself.
